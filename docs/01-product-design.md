# 产品设计：HTML Collab（暂定名）

> 一句话定位：**Agent 产出的 HTML 的协作评审层。** 把本地 HTML 一键在线化，协作者在网页上选中元素块留评论，评论结构化后回流给 agent，成为下一轮迭代的上下文。

---

## 1. 背景与判断

- Agent（Claude Code / Cursor 等）正在成为内容生产的主力，而 HTML 是 agent 最自然的交付载体：报告、原型、落地页、数据看板、活动页……都是"一个 HTML 文件"。
- 但这些 HTML 的协作方式还停留在石器时代：截图 + 微信语音描述"第三屏那个蓝色按钮往左挪一点"，然后人肉翻译给 agent。
- 反观设计领域，Figma 用"选中对象 → 评论 → @人"定义了协作范式。**HTML 缺一个 Figma 式的评审层，而且这个评审层天然应该长在 agent 工作流里**——因为改 HTML 的不再是人，是 agent。

核心洞察：**评论不是给人看的，是给 agent 读的。** 所以评论必须锚定到具体 DOM 元素、必须结构化、必须能被 CLI/MCP 一条命令拉回本地，直接进 agent 的上下文。

## 2. 核心循环（Core Loop）

```
 ①创作                ②在线化               ③评审
┌──────────┐  push   ┌──────────┐  链接   ┌──────────┐
│ Claude    │ ──────▶ │ 在线页面  │ ──────▶ │ 协作者    │
│ Code 产出 │         │ (可评论)  │         │ 选元素评论 │
│ HTML      │ ◀────── │          │ ◀────── │          │
└──────────┘  pull   └──────────┘  评论   └──────────┘
 ⑤agent 带着评论上下文改 HTML，push 新版本，评论标记已解决 → 回到①
```

1. **Push**：创作者在本地用 agent 做出 `index.html`，执行 `htmlcollab push`，得到一个链接。
2. **Share**：把链接发给协作者。协作者**零安装、零注册**（填个昵称即可）打开就能看。
3. **Comment**：协作者移动鼠标，元素块高亮；点击选中，弹出评论框；评论钉在这个元素上，右侧栏汇总所有评论线程。
4. **Pull**：创作者执行 `htmlcollab pull`（或 agent 通过 MCP 工具直接读），得到结构化反馈：每条评论带着它锚定的元素、HTML 片段、讨论线程。
5. **Iterate**：agent 读着这些上下文改 HTML，push 新版本。已处理的评论自动/手动标记 resolved，页面上能看到版本历史。

## 3. 用户与场景

| 角色 | 画像 | 关键诉求 |
|---|---|---|
| 创作者 | 用 Claude Code 的开发者 / 独立开发 / 增长运营 | push 和 pull 必须一条命令；评论回来就能直接喂给 agent |
| 评审者 | 老板、客户、设计师、同事 | 点开链接就能用，不学习、不注册、不装东西 |

典型场景：
- 给客户做落地页，客户直接在页面上圈着改，不再来回传截图。
- Agent 生成的数据报告发给团队，大家在图表上直接标"这个口径不对"。
- 独立开发做产品官网，找朋友评审文案和布局。

## 4. 功能设计

### 4.1 MVP（必须有）

**发布**
- `htmlcollab push [file]`：上传 HTML，返回链接。同一目录再次 push 自动成为新版本。
- 单文件优先；内联资源直接可用，外链资源（CDN）原样保留。

**评审页（Viewer）**
- 上方工具栏：页面标题、版本切换、评论模式开关、"复制链接"。
- 中间是原 HTML 的**忠实渲染**（沙箱 iframe，不改动用户的 HTML 行为）。
- 评论模式下：hover 高亮元素块（带标签名/类名提示），点击选中 → 侧边浮出评论输入框。
- 右侧评论栏：全部线程列表，按元素分组；点击评论 → 页面滚动到对应元素并高亮。
- 评论支持回复（线程化）、resolve / reopen。
- 评审者身份：首次评论时填昵称，存 localStorage。MVP 不做账号体系。

**回流（差异化核心）**
- `htmlcollab pull`：输出结构化反馈（markdown / JSON），每条包含：元素锚点、元素 HTML 片段、评论正文、回复线程、状态。
- `htmlcollab mcp`：作为 MCP server 跑起来，Claude Code 直接调 `list_comments` / `resolve_comment` / `push_version`，无需人工中转。
- 评论状态机：`open → resolved`（人工或 agent 标记）/ `outdated`（新版本里锚点元素消失）。

**版本**
- 每次 push 产生不可变版本 v1, v2, …；评论挂在具体版本上，但在评论栏跨版本可见。
- 新版本发布时做锚点迁移：元素还在 → 评论跟过去；元素没了 → 标记 outdated。

### 4.2 v2（明确不做进 MVP，但设计上留口）

- 文本级评论（选中一段文字而非整个元素）
- 多文件站点 / 资源目录上传
- 账号体系、团队空间、权限（当前用"链接即权限"+ 所有者 token）
- 评论 @人 与通知（邮件/Slack/钉钉）
- 实时光标 / 在线状态（presence）
- 评审者直接提"建议修改"（suggest edit，类似 Google Docs 建议模式）
- 公开画廊 / 模板

### 4.3 刻意不做

- WYSIWYG 编辑器——修改由 agent 完成，这是本产品的立身之本，不和 Webflow/Framer 竞争。
- 实时协同编辑（CRDT）——编辑发生在本地 agent 侧，线上只做评审。

## 5. 关键产品决策与理由

| 决策 | 选择 | 理由 |
|---|---|---|
| 评论锚定粒度 | MVP 只做**元素级** | 元素块是 agent 能精确定位和修改的单位；文本级锚定复杂度高一个数量级，v2 再做 |
| 评审者门槛 | 无账号，昵称即可 | 评审者是被拉来的，任何注册流程都会杀死这个产品 |
| HTML 修改权 | 只读评审，修改永远走 agent | 保持"评论 = agent 的需求输入"这一心智，避免双写冲突 |
| 锚点稳定性 | push 时给元素注入稳定 ID（详见技术设计） | 让评论在 agent 改动 HTML 后仍能存活，这是体验成败的关键 |
| 分享模型 | 知道链接即可评论；所有者持 token 可管理 | MVP 最简权限模型，够用 |
| 服务形态 | CLI（npx 免安装）为主入口，HTTP API 为能力本体；MCP/Skill 只是薄适配壳 | 用户可能用任何 agent，"能跑 shell"是所有 coding agent 的最大公约数；不绑死单一生态（详见技术设计 §11） |

## 6. 北极星与验证指标

- 北极星：**完成完整循环的页面数**（push → 至少 1 条外部评论 → pull → 再 push）。
- 辅助：链接打开→首条评论的转化率（评审端易用性）；pull 后 24h 内出现新版本的比例（回流是否真的被 agent 消费）。
